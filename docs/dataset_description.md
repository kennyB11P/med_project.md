# 1. Источники и состав данных

В качестве исходного источника использовался открытый датасет отзывов покупателей маркетплейса Wildberries: nyuuzyou/wb-feedbacks.

Данные были скачаны в формате parquet с использованием начальной предобработки:

- удаление пустых отзывов;
- замена переносов строк на пробелы;
- удаление лишних пробелов;
- нормализация пробельных символов;
- обрезка слишком длинных отзывов до 5000 символов.

# 2. Базовый EDA с выводами, важными для моделирования

На этапе EDA проанализировал не только техническую структуру данных, но и смысловую структуру самих отзывов. Для этого я взял поделил данные на чанки и из каждого взял по N случайных примеров и сгруппировал их с помощью HDBSCAN. Это позволило выявить некоторые интересные паттерны в данных.

В результате кластеризации были выделены 278 кластеров. Большая часть кластеров была успешно интерпретирована и размечена. Кластеры оказались не просто абстрактными тематическими группами, а группами отзывов по сочетанию трех факторов:

- товарная категория;
- тональность отзыва;
- тип проблемы или похвалы.

Например, отдельно выделялись кластеры положительных отзывов о платьях, футболках, обуви, детских товарах, сумках, книгах, косметике и других категориях. Также выделялись проблемные кластеры: отзывы о неподходящем размере, несоответствии цвета, браке, поломках, плохом качестве материала, поврежденной упаковке, проблемах доставки и комплектации.

Важный вывод EDA состоит в том, что исходные данные сильно смещены в сторону положительных отзывов. Большая часть объектов относится к положительным или смешанным отзывам, где покупатель в целом доволен товаром, но одновременно указывает на отдельную проблему. Это означает, что простая случайная выборка из исходного датасета была бы неудачной: в ней оказалось бы слишком много положительных отзывов и недостаточно примеров редких проблемных классов.

По результатам кластерного анализа были выделены следующие крупные группы отзывов:

1. Положительные отзывы о товарах.
2. Отзывы с проблемами размера и посадки.
3. Смешанные отзывы: товар понравился, но есть конкретная претензия.
4. Жалобы на брак, дефекты и поломки.
5. Несоответствие описанию, фотографии, цвету или ожиданиям.
6. Отзывы о цене и соотношении цена/качество.
7. Нейтральные и информационные отзывы.
8. Редкие проблемные случаи: подделка, комплектация, доставка, упаковка.

Особенно важным оказался пласт смешанных отзывов. В таких отзывах покупатель может одновременно хвалить товар и указывать на проблему. Например, отзыв может содержать положительную оценку качества, но при этом жалобу на неподходящий размер, другой цвет или дефект. Это подтвердило, что задача должна решаться именно как multi-label классификация: одному отзыву может соответствовать несколько классов.

Кластеризация также показала, что один кластер не всегда соответствует одному классу. Некоторые кластеры содержали сразу несколько смысловых признаков: например, размер, качество, несоответствие описанию и положительную оценку. Поэтому кластеры использовались не как готовые классы, а как инструмент анализа данных и поиска типичных паттернов.

Основные выводы, важные для моделирования:

- исходные данные несбалансированы: положительные отзывы встречаются значительно чаще проблемных;
- простая случайная выборка не подходит для формирования обучающего датасета;
- для обучения нужно целенаправленно добирать примеры по редким классам;
- задача является multi-label, так как один отзыв может содержать несколько проблем одновременно;
- положительный отзыв не всегда означает отсутствие проблемы;
- редкие классы требуют отдельного контроля при формировании выборки;
- кластеры можно использовать как источник кандидатов для разметки и анализа ошибок;
- для оценки качества модели нужно смотреть метрики по каждому классу отдельно.

На основе EDA была выбрана контролируемая стратегия формирования выборки. Кластеризация показала, что исходные данные сильно смещены в сторону положительных отзывов, а редкие проблемные классы встречаются значительно реже. Поэтому простая случайная выборка не подходила: она дала бы много однотипных положительных примеров и мало данных по важным проблемам.

Из-за этого выборка формировалась с контролем наполнения классов: для каждого класса добирались подходящие отзывы. Также была сохранена multi-label разметка, так как EDA показал, что один отзыв часто содержит несколько смыслов одновременно, например положительную оценку и жалобу на размер, качество или несоответствие описанию.

Кроме того, результаты EDA помогли выделить наиболее полезные примеры для golden set: смешанные отзывы, редкие классы и спорные случаи на границе нескольких категорий.

# 3. Оценка качества разметки и предложения по улучшению

Сначала был вручную собран golden set, на котором оценивалось качество разметки и подбирался промпт для LLM. После этого с помощью LLM и подобранного промпта была размечена основная выборка размером 1248 (максимум 13 классов * 100, но у меня multi-label классификация) примеров: примерно по 100 примеров на каждый класс. 

Список классов:

1. Брак / дефект товара
2. Низкое качество материала
3. Проблема с размером / посадкой
4. Несоответствие описанию
5. Несоответствие ожиданиям / эффекту
6. Проблема с комплектацией
7. Проблема с упаковкой
8. Проблема доставки / получения
9. Цена / ценность
10. Подделка / оригинальность
11. Положительный отзыв
12. Нейтральный / информационный отзыв
13. Другая проблема

Качество разметки оценивалось на golden set с использованием метрик:

- precision: 0.86
- recall: 0.81
- F1-score: 0.83

Повысить качество разметки можно за счет:

- использования более мощной LLM для автоматической разметки;
- дополнительной доработки промпта;
- расширения golden set более полезными объектами: спорными примерами, редкими классами и случаями, где модель чаще ошибается;
- ручной проверки части автоматически размеченных примеров;
- отдельного контроля качества по каждому классу.

# 4. Алгоритм формирования выборки и стратегия валидации

Кластеризация показала, что в данных преобладают положительные отзывы, а проблемные классы представлены заметно слабее. Также было выявлено много смешанных отзывов, где одновременно присутствуют положительная оценка и конкретная проблема. Поэтому итоговая обучающая выборка формировалась не полностью случайно, а с контролем наполнения классов.

Алгоритм формирования выборки:

1. Берутся очищенные parquet-файлы с отзывами.
2. Данные просматриваются по чанкам, чтобы не перемешивать весь большой датасет целиком.
3. Из отзывов берется текстовое поле text.
4. По результатам кластеризации отдельно рассматривались кластеры с разными типами отзывов: положительные отзывы, отзывы с проблемами размера и посадки, смешанные отзывы, жалобы на брак и дефекты, несоответствие описанию, нейтральные отзывы и редкие проблемные случаи.
5. Кластеры с проблемами использовались как ориентир для того, какие типы отзывов нужно целенаправленно добирать в итоговую выборку.
6. Каждый выбранный отзыв подается в LLM для multi-label разметки.
7. Если отзыв относится к одному или нескольким классам, которые еще нужно добрать, он добавляется в итоговую выборку.
8. Для каждого класса набирается заданное количество примеров: на данный момент около 100.
9. Так как задача multi-label, один отзыв может засчитываться сразу в несколько классов.
10. После разметки данные сохраняются в csv и parquet.


Такая стратегия позволила избежать перекоса в сторону положительных отзывов и целенаправленно добрать примеры для редких проблемных категорий. Кроме того, результаты кластеризации помогли выделить наиболее полезные объекты для разметки: смешанные отзывы, спорные случаи и примеры на границе нескольких классов.

Стратегия валидации:

- разделить размеченный датасет на train/validation/test;
- учитывать multi-label постановку задачи при разбиении данных;
- контролировать распределение классов в каждой части выборки;
- отдельно оценивать качество по каждому классу;
- использовать precision, recall и F1-score вместо одной общей accuracy;
- дополнительно проверять качество на вручную собранном golden set.

Финальный датасет предназначен для обучения модели multi-label классификации отзывов покупателей по типам проблем.
