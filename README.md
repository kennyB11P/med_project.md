# 1. Бизнес-анализ / Business Understanding

## 1.1. Общее описание проекта

Проект предполагает разработку AI-ассистента для врачей в формате цифрового сервиса, который помогает врачу быстрее анализировать медицинскую информацию, сопоставлять симптомы с клиническими данными и получать справочные рекомендации на основе проверенных медицинских источников.

Сервис не заменяет врача и не принимает окончательные медицинские решения самостоятельно. Его роль — выступать в качестве инструмента поддержки принятия врачебных решений. Окончательная постановка диагноза, назначение лечения и ответственность за медицинское решение остаются за врачом.

Предполагается, что сервис будет состоять из нескольких функциональных разделов:

1. Раздел анализа медицинских изображений: рентген, КТ, МРТ, УЗИ и другие типы снимков. На данный момент конкретный выбор не сделан
2. Раздел интеллектуальной поддержки диагностики: врач вводит описание состояния пациента, жалобы, анамнез, результаты анализов, после чего система с помощью RAG-подхода ищет релевантную информацию в медицинской базе знаний и предлагает возможные диагностические гипотезы.
3. Раздел содержащий основную информацию по каждому пациенту: в каждую встречу, собирая полную информацию жалоб пациента, модель делает summary, который заносится в историю для данного пациента. В нее же заносятся препараты, которые ему прописали. Таким образом для каждого пользователя будет история его состояния, которую так же можно анализировать моделью.

## 1.2. Организационная структура проекта

Ключевые участники со стороны заказчика:

**1. Руководитель проекта со стороны медицинской организации**

Отвечает за общую координацию проекта, согласование целей, сроков, бюджета и приоритетов. Принимает основные управленческие решения и контролирует, чтобы сервис соответствовал интересам организации.

**2. Главный врач или медицинский директор**

Определяет медицинскую ценность продукта, утверждает область применения сервиса и контролирует, чтобы система не противоречила клинической практике. Также участвует в определении ограничений: какие рекомендации может давать ассистент, а какие решения должны оставаться исключительно за врачом. То есть "Руководитель проекта со стороны медицинской организации" необязательно главный врач.

**3. Врачи-эксперты предметной области**

Это основные консультанты по медицинской части. Например, если первый модуль связан с анализом снимков, в проекте должны участвовать врачи-рентгенологи. Если модуль связан с описанием симптомов и диагностической поддержкой, необходимы терапевты, пульмонологи, кардиологи или другие профильные специалисты.

Их задачи:

- описать реальные рабочие сценарии;
- предоставить примеры типовых случаев;
- оценить качество ответов ассистента;
- указать на медицинские ошибки и недопустимые формулировки;
- участвовать в тестировании MVP.

**4. Основные пользователи сервиса**

Основными пользователями будут врачи, которые работают с пациентами и медицинскими данными. В зависимости от выбранного MVP это могут быть:

- врачи-рентгенологи;
- терапевты;
- врачи приёмного отделения;
- узкие специалисты;
- ординаторы и младшие врачи под контролем старших специалистов.

Вторичными пользователями могут быть заведующие отделениями, медицинские администраторы и сотрудники контроля качества, которым важно отслеживать эффективность работы врачей и качество медицинской документации.

**5. ИТ-отдел медицинской организации**

Отвечает за интеграцию сервиса с существующими системами: медицинской информационной системой, PACS/RIS для снимков, электронными медицинскими картами, внутренними базами данных и системами авторизации.

**6. Специалист по информационной безопасности**

Контролирует защиту персональных и медицинских данных, права доступа, хранение логов, шифрование, разграничение ролей и соответствие требованиям по обработке чувствительной информации.

**7. Юрист или специалист по комплаенсу**

Оценивает правовые риски использования AI в медицине, формулировки пользовательских соглашений, ограничения ответственности, требования к хранению данных и необходимость сертификации решения.

**8. Команда разработки**

Со стороны исполнителя в проекте участвуют:

- product manager или бизнес-аналитик;
- MLOps-инженер;
- UX/UI-дизайнер;
- QA-инженер;
- специалист по разметке и подготовке медицинских данных.
- На раннем этапе все эти должности занимает один человек

## 1.3. Рабочая коммуникация и сбор контактов

На старте проекта необходимо сформировать рабочую группу и определить каналы коммуникации.

Для этого создаются:

1. Основной рабочий чат для оперативных вопросов.
2. Отдельный чат или канал для медицинских экспертов.
3. Регулярные встречи с заказчиком, например один раз в неделю.
4. Отдельные встречи с врачами для сбора требований и проверки гипотез.
5. Общее хранилище документов: протоколы встреч, требования, схемы процессов, список рисков, медицинские источники, тестовые сценарии.
6. Таблица контактных лиц с зонами ответственности.

## 1.4. Бизнес-цель проекта

**Создать AI-сервис для врачей, который сокращает время анализа медицинской информации, повышает полноту врачебного заключения и помогает врачу быстрее находить обоснованные диагностические гипотезы на основе медицинских данных и проверенных источников.**

Дополнительные бизнес-цели:

1. Сокращение времени на подготовку медицинского заключения.
2. Снижение нагрузки на врачей при работе с типовыми случаями.
3. Повышение качества и стандартизации медицинской документации.
4. Снижение риска пропуска патологий на медицинских изображениях.
5. Ускорение поиска информации по клиническим рекомендациям.
6. Повышение доступности экспертной поддержки для менее опытных врачей.
7. Повышение лояльности врачей к цифровым инструментам медицинской организации.
8. Создание технологического преимущества медицинской организации перед конкурентами.

## 1.5. Проблема, которую решает проект

Врачи ежедневно работают с большим объёмом информации: жалобы пациента, анамнез, результаты анализов, медицинские изображения, предыдущие заключения, клинические рекомендации и протоколы лечения. Из-за высокой нагрузки врач может тратить значительное время на поиск информации, оформление документов и сопоставление данных из разных источников.

Основные проблемы:

1. Высокая нагрузка на врачей.
2. Дефицит времени на подробный анализ каждого случая.
3. Риск пропуска важных признаков на снимках или в описании пациента.
4. Разрозненность медицинской информации.
5. Сложность быстрого поиска по клиническим рекомендациям.
6. Неодинаковое качество медицинских заключений у разных специалистов.
7. Большое количество рутинной работы при оформлении документов.
8. Недостаток поддержки для молодых врачей и специалистов с меньшим опытом.

AI-ассистент должен помочь врачу быстрее ориентироваться в данных, но не должен подменять собой клиническое мышление специалиста.

## 1.6. Существующие решения и их ограничения

На рынке уже существуют решения для применения искусственного интеллекта в медицине. Их можно разделить на несколько групп.

**1. Системы анализа медицинских изображений**

Такие решения помогают обнаруживать признаки патологий на КТ, МРТ, рентгеновских снимках и других изображениях. Они могут использоваться для подсветки подозрительных областей, сортировки исследований по срочности и помощи врачу-рентгенологу при подготовке заключения.

Ограничения таких решений:

- часто они ориентированы только на один тип снимков или одну группу заболеваний;
- не всегда интегрированы с внутренними медицинскими системами конкретной клиники;
- могут быть дорогими для внедрения;
- требуют качественной валидации на локальных данных;
- не всегда объясняют врачу, на основании чего был сделан вывод.

**2. Клинические справочные системы**

Такие системы помогают искать информацию по заболеваниям, лекарствам, клиническим рекомендациям и протоколам лечения.

Ограничения:

- врач сам должен формулировать запрос и искать нужный раздел;
- система не всегда учитывает конкретный контекст пациента;
- поиск может быть неудобен при высокой нагрузке;
- информация может быть представлена в виде длинных документов, которые нужно читать вручную.

**3. LLM-ассистенты общего назначения**

Большие языковые модели могут объяснять медицинские термины, помогать структурировать информацию и формировать черновики текстов.

Ограничения:

- недостаточная интеграция с рабочим процессом врача!!!
- риск галлюцинаций;
- отсутствие гарантированной опоры на утверждённые медицинские источники;
- проблемы с обработкой персональных медицинских данных;
- отсутствие медицинской сертификации;
- невозможность использовать такие модели как самостоятельный диагностический инструмент;

**4. Внутренние медицинские информационные системы**

В клиниках уже могут использоваться медицинские информационные системы, электронные карты пациентов, PACS/RIS-системы для хранения снимков и модули документооборота.

Ограничения:

- такие системы чаще хранят данные, но не помогают интеллектуально их анализировать;
- поиск по данным может быть неудобным;
- нет полноценной поддержки диагностических гипотез;
- нет автоматического сопоставления симптомов, снимков и клинических рекомендаций.

Таким образом, существующие решения закрывают отдельные части задачи, но не всегда дают врачу единый рабочий инструмент, который объединяет анализ снимков, поиск по медицинским источникам, работу с описанием пациента и подготовку медицинских документов.

## 1.7. Чем предлагаемое решение отличается от существующих

Предлагаемый AI-ассистент должен отличаться не тем, что он “заменяет врача”, а тем, что он встраивается в реальный рабочий процесс врача и объединяет несколько полезных функций в одном сервисе.

Ключевые отличия:

1. Единый интерфейс для нескольких сценариев: снимки, симптомы, документы, справка.
2. Использование RAG-подхода, при котором ответы формируются с опорой на проверенную базу медицинских источников.
3. Отображение источников, на основании которых система дала рекомендацию.
4. Контроль врача на каждом этапе.
5. Возможность интеграции с медицинскими информационными системами клиники.
6. Аудит действий: сохранение запроса, ответа, источников и финального решения врача.
7. Настройка под конкретное медицинское направление и локальные клинические регламенты.
8. Возможность начать с узкого MVP и постепенно расширять систему!!!

## 1.8. Предлагаемый скоуп MVP

Чтобы разработать универсального AI-врача потребуется больше времени, чем один месяц, поэтому MVP должен быть ограничен конкретным сценарием, чтобы можно было проверить пользу продукта, качество модели и готовность врачей использовать сервис.

MVP:

**AI-ассистент для врача, который работает с описанием состояния пациента и помогает найти релевантные диагностические гипотезы на основе клинических рекомендаций и медицинской базы знаний - реализация с помощью RAG. Для каждого пациента формируется история болезни, содержащая summary его состояний на каждый прием врача (того или иного. Например у пациента проблема по части ЛОРа, но также пациент год назад проходил обследование у гастроэнтеролога - что может быть полезно для поиска причины ЛОР заболевания), а также история употребленных лекарств. Модуль анализа медицинских изображений, с прицелом на расширение мощностей и улучшение модели.**

Функции MVP:

1. Ввод описания пациента: жалобы, анамнез, симптомы, результаты анализов.
2. Поиск релевантных фрагментов в медицинской базе знаний с помощью RAG.
3. Формирование списка возможных диагностических гипотез.
4. Указание аргументов “за” и “против” каждой гипотезы.
5. Ссылки на использованные источники.
6. Предупреждение о тревожных симптомах, требующих срочного внимания.
7. Формирование черновика заключения или краткого резюме для врача.
8. Возможность врачу оценить ответ: полезно / не полезно / ошибка.
9. Модуль анализа медицинских изображений

## 1.9. Основные пользователи

Основной пользователь — врач, который принимает клиническое решение.

Портрет пользователя:

- работает с большим количеством пациентов или исследований;
- испытывает дефицит времени;
- регулярно обращается к медицинским рекомендациям;
- оформляет медицинские документы;
- нуждается в быстром доступе к структурированной информации;
- не хочет, чтобы система принимала решение за него, но готов использовать её как помощника.

Для MVP основным пользователем можно выбрать врача-терапевта или врача приёмного отделения, если акцент делается на анализ описания состояния пациента. Если акцент делается на снимках, основным пользователем будет врач-рентгенолог.

## 1.10. Критерии успеха проекта

бизнес-метрики:

1. Сокращение среднего времени подготовки заключения.
2. Увеличение количества обработанных случаев без роста нагрузки на врача.
3. Доля врачей, которые регулярно используют сервис.
4. Уровень удовлетворённости врачей.
5. Снижение количества неполных или плохо структурированных заключений.
6. Снижение времени поиска информации по клиническим рекомендациям.

продуктовые метрики:

1. Количество активных пользователей в неделю.
2. Количество запросов к ассистенту.
3. Доля ответов, которые врач отметил как полезные.
4. Доля ответов, которые врач исправил.
5. Среднее время ответа системы.
6. Количество переходов к источникам.

ML-метрики:

1. Точность извлечения релевантных источников в RAG.
2. Полнота найденной информации.
3. Доля ответов без неподтверждённых утверждений.
4. Чувствительность и специфичность для модуля анализа изображений.
5. Количество критических ошибок на тестовом наборе.
6. Согласованность ответов с экспертной оценкой врачей.
